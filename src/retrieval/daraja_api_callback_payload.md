# Safaricom Daraja API — C2B & B2C Callback Payload Specification

## 1. API Context

The Safaricom Daraja API is the programmatic interface to the M-Pesa mobile money system. The Lipa Na M-Pesa Online (STK Push) C2B flow and the B2C payment/disbursement flow both deliver transaction results to a merchant-registered callback URL via HTTPS POST with a JSON payload. This document defines the structural and validation rules for processing these callback payloads.

## 2. C2B STK Push Callback — Payload Structure (Body.stkCallback)

The top-level JSON object for a C2B STK Push response callback contains a single envelope key `Body`, which wraps an `stkCallback` object:

```json
{
  "Body": {
    "stkCallback": {
      "MerchantRequestID": "29130-4824315-1",
      "CheckoutRequestID": "ws_CO_1611692021_93827483",
      "ResultCode": 0,
      "ResultDesc": "The service request is processed successfully.",
      "CallbackMetadata": {
        "Item": [
          { "Name": "MpesaReceiptNumber", "Value": "QWE23RT4Y7", "ValueType": "string" },
          { "Name": "TransactionDate", "Value": 20250125163000, "ValueType": "int" },
          { "Name": "PhoneNumber", "Value": 254712345678, "ValueType": "int" },
          { "Name": "Amount", "Value": 1500.00, "ValueType": "float" }
        ]
      }
    }
  }
}
```

### Key reference table

| Key | Parent | Type | Required | Description |
|-----|--------|------|----------|-------------|
| MerchantRequestID | stkCallback | string | Yes | Unique request identifier generated by merchant; used for idempotency matching |
| CheckoutRequestID | stkCallback | string | Yes | Safaricom-generated checkout identifier; used for transaction lookup |
| ResultCode | stkCallback | int | Yes | 0 = success, any non-zero = transaction failure (see error code table) |
| ResultDesc | stkCallback | string | Yes | Human-readable result description |
| CallbackMetadata.Item | stkCallback | array | Only on ResultCode=0 | Array of key-value metadata pairs describing the completed transaction |
| MpesaReceiptNumber | Item entry where Name="MpesaReceiptNumber" | string | Yes (on success) | Alphanumeric transaction receipt code; globally unique key |
| TransactionDate | Item entry where Name="TransactionDate" | int | Yes (on success) | Transaction timestamp in YYYYMMDDHHmmss format |
| PhoneNumber | Item entry where Name="PhoneNumber" | int | Yes (on success) | Customer MSISDN (E.164 without leading +) |
| Amount | Item entry where Name="Amount" | float | Yes (on success) | Transaction amount in KES |
| TransactionType | stkCallback | string | No (meta) | Derived from MerchantRequestID prefix or lookup table |

## 3. B2C Payment Callback — Payload Structure (ResultParameters)

The B2C (Business-to-Customer) result callback uses a different structure under `Result.Parameters` rather than `Body.stkCallback`:

```json
{
  "Result": {
    "ResultType": 0,
    "ResultCode": 0,
    "ResultDesc": "The service request is processed successfully.",
    "OriginatorConversationID": "16773-3233291-4",
    "ConversationID": "AG_20250125_000000123456",
    "TransactionID": "QWE23RT4Y7",
    "ResultParameters": {
      "ResultParameter": [
        { "Key": "TransactionAmount", "Value": 5000.00 },
        { "Key": "TransactionReceipt", "Value": "QWE23RT4Y7" },
        { "Key": "B2CRecipientRegisteredMetadata", "Value": "RECEIVED" },
        { "Key": "B2CChargesPaidAccount", "Value": -45.00 },
        { "Key": "B2CUtilityAccountAvailableFunds", "Value": 950000.00 },
        { "Key": "B2CWorkingAccountAvailableFunds", "Value": 500000.00 },
        { "Key": "B2CInitiatorName", "Value": "testAPI" },
        { "Key": "B2CPartyName", "Value": "254712345678 - John Doe" }
      ]
    }
  }
}
```

### B2C Key reference

| Key | Parent Array | Type | Required | Description |
|-----|--------------|------|----------|-------------|
| TransactionID | Result | string | Yes | Safaricom-generated unique transaction ID |
| OriginatorConversationID | Result | string | Yes | Merchant's original conversation ID for idempotency |
| TransactionAmount | ResultParameter where Key="TransactionAmount" | float | Yes | Disbursed amount in KES |
| TransactionReceipt | ResultParameter where Key="TransactionReceipt" | string | Yes | M-Pesa receipt number; alphanumeric unique key |
| B2CRecipientRegisteredMetadata | ResultParameter | string | No | Status of recipient registration (RECEIVED/REJECTED/UNKNOWN) |
| B2CChargesPaidAccount | ResultParameter | float | No | Charges deducted from utility account (negative value) |
| B2CUtilityAccountAvailableFunds | ResultParameter | float | No | Remaining utility account balance |
| B2CWorkingAccountAvailableFunds | ResultParameter | float | No | Remaining working account balance |
| B2CPartyName | ResultParameter | string | No | Recipient identifier (MSISDN - Name) |

## 4. B2C Transaction Types and Result Aggregation

The B2C API supports four transaction types identified by the InitiatorCommandID field in the original request, each with distinct callback semantics:

| CommandID | Purpose | Success Code | Recipient Metadata on Success |
|-----------|---------|-------------|------------------------------|
| SalaryPayment | Disbursement of salary | 0 | "RECEIVED" |
| BusinessPayment | Payment for goods/services | 0 | "RECEIVED" |
| PromotionPayment | Promotional/advertising payment | 0 | "RECEIVED" |
| Reversal | Reversal of a previous transaction | 0 | N/A — no recipient metadata |

The system must correlate the incoming callback's InitiatorCommandID (derived from the original request context, not the callback itself) to determine the proper table routing.

## 5. Error Result Codes and Descriptions

| ResultCode (C2B/B2C) | Description | Handling |
|----------------------|-------------|----------|
| 0 | Success | Process transaction normally |
| 1 | Internal Safaricom error (insufficient balance / system error) | Trigger retry with exponential backoff; max 3 attempts |
| 2 | Service timeout | Trigger retry; log as network anomaly |
| 103 | Request cancelled by user | Log as user-abandoned; no retry |
| 1032 | Transaction rejected by user on STK popup | Log as user-rejected; no retry |
| 2001 | Invalid data in request payload | Flag for developer review; invalidate cache |
| 2002 | Bad request (malformed JSON) | Flag for developer review; log full payload |
| 2003 | MSISDN not registered for M-Pesa | Return user-friendly error; no retry |
| 2004 | Invalid transaction amount (below KES 1.00 or above KES 150,000) | Reject; flag as compliance violation |
| 2005 | Duplicate MerchantRequestID | Log as idempotency hit; return cached result if available |
| 400–407 | Various authentication/credential errors | Rotate API keys; alert operations team |

## 6. Critical Field Validation Criteria

### 6.1 MpesaReceiptNumber / TransactionReceipt

- **Type:** string (strict)
- **Format:** Exactly 10 uppercase alphanumeric characters (e.g., `QWE23RT4Y7`)
- **Uniqueness:** Must be globally unique per transaction. Any system receiving a duplicate receipt number within a 90-day rolling window MUST escalate as a possible replay attack or cache-poisoning vector.
- **Pattern regex:** `^[A-Z0-9]{10}$`
- **Null/empty handling:** If `MpesaReceiptNumber` is missing, null, or empty, the system MUST reject the entire callback and NOT process the Amount or update any balance. Log as CRITICAL integrity failure.

### 6.2 TransAmount / Amount / TransactionAmount

- **Type:** float (strictly positive)
- **Format:** Numeric value with up to 2 decimal places (e.g., `1500.00`, `50.50`, `1.00`)
- **Range:** Minimum KES 1.00 (floor); maximum KES 150,000 (per-transaction ceiling per CBK regulation, or KES 500,000 for B2C disbursements)
- **Validation rule:** Must satisfy `amount > 0.0`. Zero or negative values are invalid and MUST be rejected as data integrity failures.
- **Precision:** Parse as float, then round to 2 decimal places. Values with more than 2 decimal places must be truncated (not rounded) to avoid inflation attacks.
- **Null handling:** If Amount is null, missing, or non-numeric, reject the entire callback and flag as potential tampering.

### 6.3 MSISDN / PhoneNumber

- **Type:** string or int (coerce to string for validation)
- **Format:** E.164 for Kenya: `254XXXXXXXXX` where X is a digit (e.g., `254712345678`)
- **Length:** Exactly 12 digits starting with `254`
- **Pattern regex:** `^254[17]\d{8}$` (Safaricom prefixes: 2547XX, 2541XX)
- **Country code enforcement:** Must start with `254`. Any MSISDN without the 254 prefix (e.g., `0712345678` or `712345678`) MUST be normalized by stripping leading `0` and prepending `254`. If normalization fails or resulting length != 12, reject the callback.
- **Null handling:** Missing or null MSISDN is a CRITICAL failure — reject and log.

### 6.4 TransactionDate

- **Type:** int (coerce to string for parsing)
- **Format:** `YYYYMMDDHHmmss` (14 digits, e.g., `20250125163000`)
- **Validation:** Must be a valid datetime. Dates in the future (beyond 5 minutes clock skew) MUST be rejected as potential replay attacks.
- **Null handling:** If TransactionDate is missing or unparseable, log as integrity warning; transaction may still be processed if MpesaReceiptNumber, Amount, and MSISDN are valid (the system timestamp is used as fallback).

### 6.5 ResultCode

- **Type:** int
- **Values:** 0 (success) or non-zero (failure). For C2B, non-zero result codes will have NO CallbackMetadata.Item array — the system MUST NOT attempt to parse metadata fields from a failed transaction callback.
- **Validation:** If ResultCode is missing or null, treat as indeterminate — reject callback and place in manual review queue.

### 6.6 MerchantRequestID / OriginatorConversationID

- **Type:** string
- **Format:** Merchant-generated; typically `{timestamp}-{random}` or a UUID
- **Uniqueness:** The merchant system MUST enforce uniqueness. A duplicate MerchantRequestID within the idempotency window (typically 24 hours) MUST return the cached result for the original transaction and NOT process a duplicate.
- **Null handling:** If MerchantRequestID is missing or null, the callback cannot be correlated to any originating request. The transaction SHOULD be logged and flagged for manual reconciliation but MAY still be processed if CheckoutRequestID and MpesaReceiptNumber are present.

## 7. Edge Cases and Security Exception Logic

### 7.1 Null TransactionType Handling

The `TransactionType` field is not present in the callback payload itself. It is derived from the original API request context (STK Push vs. B2C vs. Reversal). A downstream system that attempts to read `TransactionType` from the callback directly will receive a null/undefined value.

**Detection rule:** If `TransactionType` is coerced from an external lookup (e.g., a database JOIN on MerchantRequestID) and returns null, the system MUST NOT proceed with transaction processing. This indicates either:
- A cache-poisoning attempt where the callback payload does not match any known request
- A replay attack where an attacker resubmits a callback from a different transaction type context

**Action:** Reject callback, escalate to security operations within 15 minutes.

### 7.2 Missing MerchantRequestID (Cache-Poisoning Vector)

A callback arriving without a `MerchantRequestID` (or with a `MerchantRequestID` that does not match any pending request in the system) presents a cache-poisoning risk:

- An attacker who observes the transaction flow can craft a forged callback with a valid-looking `MpesaReceiptNumber` and `Amount` but no matching `MerchantRequestID`.
- The system's idempotency cache will not recognize the request, and if the validation logic skips the `MerchantRequestID` correlation step, the forged transaction may be accepted.

**Detection rule:** Every callback MUST be correlated against a pending request record by either `MerchantRequestID` or `CheckoutRequestID`. If neither matches a known pending request:
1. Log as `CACHE_POISONING_ATTEMPT`
2. Do NOT process Amount or update balance
3. Retain the raw payload for forensic analysis
4. Alert security operations immediately

### 7.3 Phantom Duplicate Receipt Number (Replay Attack)

A callback where `MpesaReceiptNumber` already exists in the transaction log at any status (success, failed, or pending) within the last 90 days:

- **Legitimate case:** Idempotent retry of the same callback (same MerchantRequestID, same CheckoutRequestID). Return cached result; do NOT process.
- **Attack case:** Different MerchantRequestID or CheckoutRequestID but same MpesaReceiptNumber. This is a replay attack vector.

**Detection rule:**
```python
WHEN (duplicate_mpesa_receipt WITHIN 90_days) AND 
     (merchant_request_id != cached_merchant_request_id)
THEN FLAG = "REPLAY_ATTACK"
  ACTION: Discard callback, alert SOC, block initiating MSISDN
```

### 7.4 Amount Mismatch (Callback vs. Request)

The `Amount` in the callback must match the `Amount` that was originally sent in the STK Push request. A mismatch (even by KES 1.00) indicates:

- A man-in-the-middle attack modifying the payload in transit
- A merchant-side bug where the wrong amount was referenced during reconciliation

**Detection rule:**
```python
WHEN (callback_amount != request_amount) AND (abs_diff >= 1.0)
THEN FLAG = "AMOUNT_INTEGRITY_FAILURE"
  ACTION: Do NOT credit customer account; hold in suspense ledger
  ACTION: Escalate to finance team within 1 hour
  EXCEPTION: B2C charges (B2CChargesPaidAccount) may be negative; do NOT apply Amount Integrity check to charge fields
```

### 7.5 Callback on Non-Existent CheckoutRequestID

A callback with a `CheckoutRequestID` that the system has no record of requesting:

- Possible causes: delayed callback from a previous environment/test, attacker injecting synthetic callbacks, or cross-account ID collision.
- Safaricom guarantees `CheckoutRequestID` uniqueness per environment (sandbox vs. production). Any callback with an unknown `CheckoutRequestID` in production MUST be treated as suspicious.

**Detection rule:** Reject callback if `CheckoutRequestID` is not found in the transaction request log. Log as `ORPHAN_CALLBACK`.

### 7.6 B2C ResultCode 1 with Partial Success Payload

A known edge case: B2C ResultCode=1 (insufficient balance / system error) has been observed returning partial `ResultParameters` that include `TransactionReceipt` and `TransactionAmount` but with negative `B2CChargesPaidAccount` values that do not reconcile.

**Detection rule:** If `ResultCode != 0` but `TransactionReceipt` is present:
- Log as `INCONSISTENT_B2C_STATE`
- Do NOT credit the recipient
- Place funds on hold in suspense until manually reconciled
- Flag anomaly report for CBK compliance audit

### 7.7 Negative Amount Values

`B2CChargesPaidAccount` is the only field where negative values are expected (charges deducted from merchant). Any other amount field (`Amount`, `TransactionAmount`) with a negative or zero value MUST be rejected.

**Allowlist for negative values:** Only `"B2CChargesPaidAccount"`, `"B2CWorkingAccountAvailableFunds"` delta. All other numeric fields: strictly positive enforcement.

## 8. Payload Sanitization Rules

Before any transaction processing occurs, the callback payload MUST pass through the following sanitization pipeline:

| Step | Rule | Purpose |
|------|------|---------|
| 1 | Parse JSON; reject if malformed | Prevent injection via malformed payload |
| 2 | Strip BOM (Byte Order Mark) if present | Handle encoding edge cases |
| 3 | Drop unexpected top-level keys beyond Body/Result | Prevent schema poisoning |
| 4 | Reject if Content-Type != application/json | MIME type enforcement |
| 5 | Validate HMAC signature (if enabled) | Authenticity verification |
| 6 | Coerce all numeric fields according to section 6 types | Type safety |
| 7 | Reject if any critical field (MpesaReceiptNumber, Amount, MSISDN) fails validation | Data integrity gate |
| 8 | Correlate against pending request store | Idempotency + anti-replay |
| 9 | Apply MSISDN E.164 normalization | Consistent addressing |
| 10 | Log raw payload to audit store BEFORE processing | Forensic audit trail |

## 9. Rate Limiting and Throttling

Safaricom Daraja API enforces the following rate limits on requests to the API (not callbacks, but relevant for retry logic):

| Endpoint | Limit | Window |
|----------|-------|--------|
| STK Push (Lipa Na M-Pesa Online) | 10 requests per second | Rolling 1-second window |
| B2C Payment Request | 5 requests per second | Rolling 1-second window |
| Query Status | 10 requests per second | Rolling 1-second window |
| OAuth Token Generation | 1 request per 55 seconds | Sliding window (token TTL = 3600s) |

Callback receivers (merchant servers) should implement their own rate limiting to handle bursts: Safaricom may deliver up to 50 callbacks per second during peak hours (e.g., salary disbursement days).
