M-Pesa SHAP Credit Scoring — Model Specification

1. OVERVIEW

Model Name: sentinai_credit_shap_001
Model Type: SHAP-driven XGBoost creditworthiness scorer
Purpose: Generate explainable credit risk scores for M-Pesa wallet holders (unbanked and underbanked segments in Kenya) using alternative transaction data. Scores support microloan origination (KES 1,000–KES 200,000) with full regulatory explainability under CBK Credit Reference Bureau (CRB) guidelines.

Development Framework: XGBoost 2.0.3 with SHAP 0.45.0 explainability layer
Target Variable: Binary default indicator (1 if loan repayment >30 days overdue)
Training Period: January 2024 – December 2025
Jurisdiction: Kenya (KE)

2. MODEL INPUTS — ALTERNATIVE CREDIT FEATURES

All features derived exclusively from M-Pesa transaction history (no traditional CRB data required):

Category 1 — Cash Flow Stability (6 features)
- avg_monthly_income_6m: Mean incoming P2P and deposit volume over 6 months (KES)
- income_volatility_cv: Coefficient of variation of monthly income (lower = more stable)
- min_monthly_balance_6m: Lowest end-of-month wallet balance over 6 months (KES)
- savings_rate: Ratio of end-of-month balance to total incoming flows
- income_growth_trend: Linear regression slope of monthly income over 6 months
- txn_regularity_score: Standard deviation of inter-transaction intervals (inverse)

Category 2 — Repayment History (5 features, proxy behavior)
- fuliza_repayment_rate: Proportion of Fuliza (M-Pesa overdraft) advances fully repaid within due date
- loan_utilization_ratio: Total loan amount drawn / total loan limit approved
- avg_days_to_repay: Mean repayment lag for Fuliza and previously disbursed loans
- max_delinquency_days: Longest observed delinquency period in days
- credit_resurrection_count: Number of times account returned to good standing after delinquency

Category 3 — Transaction Behavior (7 features)
- avg_txn_amount: Mean transaction value over 90 days (KES)
- txn_frequency_per_day: Mean number of transactions per day
- merchant_diversity: Number of unique merchant categories transacted with in 90 days
- airtime_to_cash_ratio: Ratio of airtime purchases to cash withdrawals (low ratio = liquidity constrained)
- savings_group_indicators: Frequency and volume of group/chama contribution transactions
- international_remittance_inflow: Proportion of income from international remittance (KES)
- utility_payment_regularity: Standard deviation of utility bill payment intervals

Category 4 — Digital Footprint (3 features)
- wallet_age_days: Days since first M-Pesa registration
- sim_tenure: Months with same SIM registration
- feature_phone_indicator: 1 if primary device is feature phone, 0 if smartphone

Category 5 — Social Graph (3 features derived from transaction network)
- network_density: Number of unique counterparties transacted with per month
- reciprocity_score: Ratio of incoming to outgoing P2P transactions (near 1.0 = balanced)
- trusted_network_size: Number of counterparties with >6 months of continuous transaction history

Total: 24 features

3. MODEL ARCHITECTURE AND HYPERPARAMETERS

Primary model: XGBoost Classifier (probability calibration via Platt scaling)
Base: XGBoost 2.0.3 with custom asymmetric loss function (false negative cost = 3x false positive cost)

Hyperparameters:
- objective: binary:logistic (with custom asymmetric loss weight = 3.0 for FN)
- n_estimators: 350
- max_depth: 6
- learning_rate: 0.03
- subsample: 0.75
- colsample_bytree: 0.65
- min_child_weight: 8
- gamma: 0.2
- reg_lambda: 2.0
- reg_alpha: 1.0
- scale_pos_weight: 2.5
- eval_metric: AUC-PR
- early_stopping_rounds: 50

SHAP explainability layer:
- TreeExplainer (exact SHAP values, not interventional)
- Expected value (baseline): 0.072 (mean default probability in population)
- Output: per-feature SHAP contribution values normalized to probability space

6. SHAP EXPLANABILITY — PER-APPLICATION OUTPUT

Every credit application is output with:
- base_value: 0.072 (population baseline default probability)
- f_x: Final model score (default probability in range 0.0–1.0)
- shap_values: Dict of 24 feature -> contribution value
- top_3_positive_factors: The 3 features driving score upward (toward approval) with their SHAP magnitudes
- top_3_negative_factors: The 3 features driving score downward (toward rejection) with their SHAP magnitudes
- recommendation: APPROVE / REFER / DECLINE based on threshold
- reason_codes: Regulatory-friendly reason codes mapped from SHAP (6 max)

Threshold structure:
- >=0.14: DECLINE (high default probability)
- 0.09–0.139: REFER (manual underwriting review)
- <0.09: APPROVE

7. PRODUCTION DEPLOYMENT

Inference pipeline:
1. Feature extraction from M-Pesa transaction log (90-day lookback)
2. Preprocessing: missing value imputation (median for continuous, mode for categorical), z-score normalization using training set parameters
3. XGBoost inference: predict_proba -> default probability
4. SHAP TreeExplainer: compute 24 SHAP values per application (batch = 1000, cached explainer)
5. Post-processing: Map SHAP values to reason codes, generate recommendation, log to audit store
6. API response: Return score + SHAP explanation + reason codes in JSON

Latency budget:
- Feature extraction: <200ms per application
- XGBoost inference: <5ms per application
- SHAP TreeExplainer: <50ms per application (batch)
- End-to-end: <300ms

8. FAIRNESS AND REGULATORY COMPLIANCE

CBK Credit Reference Bureau Guidelines compliance:
- No direct use of gender, ethnicity, religion, or political affiliation (verified via SHAP dependence testing)
- No geographic redlining: 47 counties equally represented in training data
- Alternative data must be explainable: SHAP satisfies CBK right-to-explanation requirement (CRB regulations Section 12)

Fairness metrics across demographic proxies:
